Lab 2: AI Intelligence — Prompts, Parsers, and Multi-Agent Orchestration

Fix broken AI behavior through prompt engineering, output parsers, tool descriptions, and multi-agent routing

Skills practiced: Diagnosing broken AI behavior through prompt engineering, output parser configuration, tool description clarity, and multi-agent routing

Prerequisites: n8n running locally, Lesson 2 talk completed, Gemini API key configured in n8n. For exercise 8: access to an LLM (ChatGPT, Claude, Gemini, or any other)

The Scenario

Same colleague, different mess. This time they left behind AI-powered workflows. The nodes are wired correctly, the connections look fine, and n8n doesn't throw any errors. But the AI behaves wrong: it classifies tickets into categories nobody asked for, it picks the wrong tool, it routes tasks to the wrong agent. The bugs aren't in the plumbing — they're in the intelligence layer. Your job: figure out why the AI is misbehaving and fix the prompts, parsers, and descriptions that control it.

Pick the exercises that interest you. They're independent and you can do them in any order.

5

Fix the AI Classifier

Approachable Structured output, output parsers, prompt-to-downstream alignment

The Challenge

Import exercise_5_structured_output.json from the workflows/ folder. This workflow takes a support ticket, sends it to an AI model for classification, parses the AI's response into structured JSON, and then routes the ticket based on urgency.

⬇ Download exercise_5_structured_output.json

When you run it, the routing will fail. The AI will respond, but the downstream nodes can't use the response correctly. Three issues:

  1. The AI prompt doesn't tell the model what format to respond in
  2. The field names in the AI's output don't match what the IF node is checking
  3. The prompt doesn't constrain the AI's output values, so the IF condition can't reliably match

Acceptance Criteria

  • The Classify Ticket node produces valid structured JSON (not free-form text)
  • The IF node correctly routes urgent tickets to "Escalate" and non-urgent to "Queue"
  • The Escalate and Queue nodes display meaningful summary text
  • You can explain all three issues and why each fix was necessary

Hints

Hint 1: Why structured output parsing fails

The Structured Output Parser expects the AI to return JSON matching a specific schema. But the AI's system prompt just says "classify it" without specifying the format. Look at the parser's jsonSchemaExample to see what fields it expects, then update the AI's prompt to explicitly request JSON output matching that schema.

Hint 2: The IF node checks a field that doesn't exist

Open the IF node and look at the condition. It checks $json.priority. Now open the Structured Output Parser and look at the schema example. What is the actual field name for urgency? The IF node needs to reference the exact field name the parser produces.

Hint 3: Unconstrained values break routing

Even if the field names match, the IF node checks for a specific string like "high". But the AI might respond with "critical", "urgent", "HIGH", or anything else. The prompt needs to tell the AI exactly which values are allowed: "urgency must be one of: high, medium, low".

6

Fix the AI Agent's Tools

Challenging Tool descriptions, system prompts, agent memory

The Challenge

Import exercise_6_agent_tools.json from the workflows/ folder. This is a chat-based AI Agent that should be able to check the weather for any city and fetch news from RSS feeds.

⬇ Download exercise_6_agent_tools.json

When you chat with it, the agent will struggle. It might refuse to use its tools, use the wrong one, or give vague responses. The wiring is correct. Three issues:

  1. Both tool descriptions are identical and vague — the agent can't tell them apart
  2. The system prompt gives the agent no guidance about its capabilities
  3. The agent has no memory, so it forgets the conversation between messages

Acceptance Criteria

  • Ask "What's the weather in Tel Aviv?" — the agent uses the Get Weather tool
  • Ask "Get me the latest tech news" — the agent uses the Get News tool
  • Ask a follow-up like "What about London?" — the agent remembers context
  • You can explain how each fix changed the agent's behavior

Hints

Hint 1: Why tool descriptions matter

When an AI Agent has multiple tools, it reads each tool's toolDescription field to decide which one to use. If both tools say "A general purpose tool," the agent has no basis for choosing. Each tool needs a clear, specific description.

Hint 2: The system prompt is too vague

The agent's system prompt is just "You are a helpful assistant." It doesn't mention weather, news, or any specific capabilities. A good system prompt tells the agent what it can do and how to use its tools.

Hint 3: Adding memory

Compare to the original (workflows/originals/6270_first_ai_agent.json). It has a "Conversation Memory" node connected to the agent. This exercise is missing that node. Add a Window Buffer Memory node and connect it to the agent's ai_memory input.

Fix the Multi-Agent System

Advanced Multi-agent delegation, tool description routing, supervisor patterns

The Challenge

Import exercise_7_multi_agent.json from the workflows/ folder. This is a Manager Agent that delegates to two specialized sub-agents: an EmailAgent and a DataAgent.

⬇ Download exercise_7_multi_agent.json

Right now, the manager only ever uses one agent, and when it does delegate, the results are wrong. Three issues that combine:

  1. The tool descriptions are swapped — EmailAgent's description says data analysis, and vice versa
  2. The manager's system prompt only mentions the EmailAgent — it never tells the manager the DataAgent exists
  3. The combined effect: the manager always delegates to the EmailAgent

Acceptance Criteria

  • Ask "Write a follow-up email to a client" — manager delegates to EmailAgent
  • Ask "Summarize key business metrics for Q3" — manager delegates to DataAgent
  • The manager correctly chooses based on the request
  • You can explain all three issues and how they interact

Hints

Hint 1: Start with the tool descriptions

Click on each sub-agent tool node and read their toolDescription fields. Then read their systemMessage fields. Do the descriptions match what the agents actually do?

Hint 2: The manager doesn't know the DataAgent exists

The ManagerAgent's system prompt says "Always delegate to the EmailAgent for any task." Even though the DataAgent is connected, the instructions tell it to ignore everything except the EmailAgent.

Hint 3: Compare with the working version

Open workflows/originals/7158_manager_agent_subagents.json and compare side by side. Look at tool descriptions, the manager's system message, and how they align.

8

Generate an AI Workflow with an LLM

Creative LLM workflow generation, AI node configuration

The Challenge

Use an LLM to generate a complete AI-powered n8n workflow as JSON. Then import it, test it, find the issues, and prompt the LLM to fix them.

Here's the spec:

"Build an n8n workflow that:
  1. Starts with a Chat Trigger (the built-in n8n chat interface)
  2. Has an AI Agent node connected to a Google Gemini model
  3. Has a Window Buffer Memory node for conversation history
  4. Has an HTTP Request Tool that calls the Open-Meteo weather API with dynamic $fromAI() expressions for latitude and longitude
  5. The agent's system prompt: "You are a weather assistant. Use the Get Weather tool with city coordinates."

Return only valid n8n workflow JSON. No explanation."

Acceptance Criteria

  • You prompted an LLM and received n8n workflow JSON
  • The JSON imports into n8n
  • You identified at least one issue
  • You prompted the LLM with the specific issue and got an updated version
  • The final workflow has a working chat interface for weather queries
  • You saved the final workflow JSON

Hints

Hint 1: Structuring your prompt for AI workflows

Be specific about node types:

  • @n8n/n8n-nodes-langchain.chatTrigger, @n8n/n8n-nodes-langchain.agent
  • @n8n/n8n-nodes-langchain.lmChatGoogleGemini, @n8n/n8n-nodes-langchain.memoryBufferWindow
  • Connection types: ai_languageModel, ai_memory, ai_tool (not main)
Hint 2: Common issues with LLM-generated AI workflows
  • Wrong connection types — connects Gemini via main instead of ai_languageModel
  • Missing $fromAI() — tool parameters are hardcoded
  • Wrong node type strings — invents names that don't exist
  • No typeVersion — every n8n node needs this field
Hint 3: Use a working workflow as a reference

Open workflows/originals/6270_first_ai_agent.json, paste it into the LLM, and say "Use this as a reference to generate a new workflow that does [your spec]. Match the structure exactly."

Fix the AI Customer Service Pipeline (Extra Hard)

Extra Hard Multi-stage AI pipeline, classifier-router alignment, cross-layer bugs

The Challenge

Import exercise_10_ai_service_pipeline.json from the workflows/ folder. This is a full AI customer service pipeline: a classifier analyzes incoming messages, a router sends them to the right specialist (refund, technical, or general), and each specialist generates a tailored response.

⬇ Download exercise_10_ai_service_pipeline.json

The architecture:

Chat → Classify Intent (LLM + Parser) → Route by Intent (Switch)
  → Refund path:    Handle Refund (LLM Chain)
  → Technical path:  Handle Technical (AI Agent + Order Lookup tool)
  → General path:    Handle General (LLM Chain)

A 13-node workflow with five bugs that span multiple layers. Test with:

  • "I want a refund for order #1234, the product arrived damaged" → Refund specialist
  • "My device keeps crashing when I try to update the firmware" → Technical specialist
  • "What are your business hours?" → General specialist

Acceptance Criteria

  • The classifier produces structured JSON with correct field names
  • The Switch node routes messages to the correct specialist
  • Refund specialist responds with refund-appropriate language
  • Technical specialist responds with consistent, factual troubleshooting
  • General specialist responds to general questions
  • You can explain all five bugs and how they interact

Hints

Hint 1: Messages always fall through to the General path

Two bugs combine. The classifier prompt says "billing", "tech_support", "other" but the Switch expects "refund" and "technical". Also, the Switch checks $json.category but the parser schema uses intent. Fix both.

Hint 2: The Refund specialist talks like tech support

The Handle Refund node's system prompt says "You are a technical support specialist." Copy-paste error. Replace with refund-appropriate language.

Hint 3: The Technical specialist's tool is mislabeled

The "Order Lookup" tool's description says "process refund requests" instead of technical troubleshooting. The agent reads this to decide when to use the tool.

Hint 4: The Technical specialist gives wild responses

Check the Gemini model's temperature: 0.9 is too high for factual technical support. Set it close to 0.

Hint 5: The five bugs summarized
  1. Classifier → Router value mismatch: "billing, tech_support, other" vs "refund, technical"
  2. Classifier → Router field mismatch: intent vs category
  3. Refund prompt: Copy-paste error — has technical support prompt
  4. Technical tool description: Describes refund instead of troubleshooting
  5. Technical temperature: 0.9 → should be ~0

Wrap-Up

If you completed exercises 5 through 7, you've built the core diagnostic skill for AI workflows: understanding that the intelligence lives in configuration, not wiring. The nodes can be perfectly connected, but the AI does the wrong thing because a prompt is vague, a description is misleading, or a field name doesn't match.

If you completed exercise 8, you've extended the LLM generation loop into the AI node ecosystem. The key insight: including a working reference workflow in your prompt dramatically improves output.

If you completed exercise 10, you've debugged the kind of multi-stage AI pipeline that actually exists in production: bugs in one layer cascade into wrong behavior in the next.
