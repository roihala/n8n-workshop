Lab 2: AI Workflows — Agents, Structure, and Collaboration

Build AI-powered workflows with structured output, tool-using agents, and multi-agent collaboration

Skills practiced: Connecting AI models as workflow nodes, structured output parsing, building AI agents with tools, multi-agent collaboration, importing and customizing advanced community workflows

Prerequisites: n8n running locally, API keys configured (Gemini or Grok), Lesson 1 completed, Lesson 2 talk completed

The Scenario

After the crash course on data flow and context management, your team is ready for the real thing: AI-powered workflows. Your manager has three requests, each one more ambitious than the last. First, build a workflow where an AI node doesn't just produce text but produces structured data that drives downstream decisions. Second, give an AI agent actual tools so it can take actions on its own. Third, build a multi-agent system where AI nodes collaborate, starting from a community workflow and adapting it rather than building from scratch. Three exercises. Let's make the AI actually useful.

1

AI Classification with Structured Output

The Challenge

Build a workflow that takes unstructured text input (a support message, a customer email, or a product review) and uses an AI node to classify it into a structured format, then routes the workflow based on that classification.

Your workflow must:

  1. Accept input (manual trigger with a hardcoded sample text, or a form trigger)
  2. Send the text to an AI node (Gemini or Grok) with a system prompt that instructs it to return structured JSON output containing at least: a category field, a priority field (high/medium/low), and a summary field
  3. Parse the AI's response into actual usable fields
  4. Route to different paths based on the priority value

Start by importing the community template at #9557 to see how a production workflow handles structured AI output and branching. Study its pattern, then build your own version.

Acceptance Criteria

  • You've imported and reviewed workflow #9557 to understand the structured output + branching pattern
  • Your own workflow sends text to an AI node and receives structured JSON back (not free-form text)
  • The AI response is parsed into individual fields that downstream nodes can reference
  • An IF node (or Switch node) routes the workflow based on the AI's classification
  • You can run the workflow with two different inputs and show it taking different paths

Hints

Hint 1: Getting structured output from the AI

The key is in the system prompt. Tell the AI exactly what JSON structure to return:

You are a support ticket classifier. Analyze the input text and return ONLY a JSON object with these fields:
- category: one of "bug", "feature_request", "question", "complaint"
- priority: one of "high", "medium", "low"
- summary: a one-sentence summary of the issue

Return ONLY the JSON object, no other text.
Hint 2: Parsing the JSON response

If the AI returns a JSON string, you need to parse it. Use a Code node with JSON.parse() on the AI's output text, or use n8n's Structured Output Parser node. The Code node approach is more reliable when the AI occasionally wraps its JSON in markdown code blocks.

Hint 3: The routing pattern

After parsing, add an IF node with the condition: {{ $json.priority }} equals "high". The "true" branch handles high-priority items, the "false" branch handles everything else. For a more sophisticated version, use a Switch node for three separate paths.

Step-by-step guide (optional)
  1. Import workflow #9557. Click through each node and study how it uses the Structured Output Parser and IF node.
  2. Create a new workflow. Add a Manual Trigger node.
  3. Add an Edit Fields node. Create a field called inputText and paste a sample support message, e.g.: "I've been trying to export my data for three days and the button just spins forever. This is blocking our entire team's quarterly report. Please fix this urgently."
  4. Add a Basic LLM Chain node. Select your AI model. Set the system prompt (see Hint 1).
  5. Set the user message to {{ $json.inputText }}.
  6. Add a Code node to parse: const parsed = JSON.parse($input.first().json.text.replace(/```json\n?/g, '').replace(/```\n?/g, '')); return [{ json: parsed }];
  7. Add an IF node. Condition: {{ $json.priority }} equals high.
  8. Add placeholder nodes on each branch (e.g., Edit Fields with action: "escalate" on true, action: "queue" on false).
  9. Test with the urgent message (should route to "true"). Then change the input to something low-priority like "Just curious, do you support dark mode?" and test again.
2

Build an AI Agent with Tools

The Challenge

Build a workflow using n8n's AI Agent node where the agent has access to at least two tools and decides which ones to use based on the input.

Start by importing #6270 — n8n's official "Build Your First AI Agent" template. Get it running, study how the agent selects tools, then extend it with your own tool or replace existing tools with ones more relevant to your work.

Your final workflow must have:

  1. A chat trigger or manual trigger
  2. An AI Agent node with a clear system prompt
  3. At least two tool nodes connected to the agent
  4. Conversation memory so the agent can reference previous messages

The test: give the agent two different inputs that should trigger different tools, and verify it chooses correctly.

Acceptance Criteria

  • Workflow #6270 ("Build Your First AI Agent") is imported and running
  • You understand how tool nodes connect to the AI Agent node
  • Your final workflow has an AI Agent with at least 2 tools and conversation memory
  • You can demonstrate the agent choosing different tools for different inputs
  • The agent's system prompt clearly defines its role and when to use each tool

Hints

Hint 1: Understanding the Agent + Tools pattern

The AI Agent node works like this: you connect tool nodes to the agent's "tools" input. Each tool has a name and description. When the agent receives a message, it reads the tool descriptions and decides which tool(s) to call. The tool descriptions are critical — if they're vague, the agent picks wrong.

Hint 2: Good tool combinations
  • HTTP Request tool (for fetching web data) + Code tool (for calculations)
  • Wikipedia tool (for factual lookups) + Calculator tool (for math)
  • Two HTTP Request tools pointed at different APIs with different descriptions

The key: tools should be clearly distinct so you can verify the agent's choice.

Hint 3: Good test inputs

If your agent has a weather tool and a calculator tool:

  • "What's the weather in Tel Aviv?" → should trigger the weather tool
  • "What's 15% tip on a 230 shekel dinner?" → should trigger the calculator tool
  • "If it's above 30 degrees in Tel Aviv, calculate how many liters of water I need for a 5km run" → might trigger both
Step-by-step guide (optional)
  1. Import workflow #6270 from n8n.io/workflows.
  2. Open the AI Agent node and study: system prompt, model, connected tools.
  3. Click on each tool node to see its name and description.
  4. Test with a message that should trigger one specific tool. Check which tool it chose.
  5. Modify the workflow: add a new tool or replace an existing one. Good options:
    • Add an HTTP Request tool pointed at a public API (e.g., https://api.exchangerate-api.com/v4/latest/USD)
    • Add a Code tool that performs a specific computation
  6. Update the agent's system prompt to mention the new tool.
  7. Add a Window Buffer Memory node if there isn't one. Connect it to the agent's "memory" input.
  8. Test with input A (tool 1) then input B (tool 2).
  9. Test with a third input referencing previous conversation: "Do that again but with different numbers."

Import and Customize a Multi-Agent Workflow

The Challenge

Import a multi-agent community workflow, get it running, understand the collaboration pattern, and customize it for a different use case.

Go to #7158 and import the "Beginner Manager Agent with Sub-Agent Tools" template. This workflow has a Manager Agent that evaluates input and delegates to specialized sub-agents (EmailAgent and DataAgent).

Your job:

  1. Import it and get it running
  2. Understand the supervisor-delegates pattern: how the manager decides which sub-agent to invoke
  3. Customize it by modifying an existing sub-agent's specialty or adding a third sub-agent
  4. Test with at least two different inputs that trigger different sub-agents

If you finish early, explore these additional multi-agent templates:

  • #9545 — Four specialist agents analyze a landing page from different angles
  • #3425 — Supervisor agent routing to three specialized data analyst sub-agents

Acceptance Criteria

  • Workflow #7158 is imported and runs
  • You can explain how the Manager Agent decides which sub-agent gets the task
  • You've customized the workflow: modified an existing sub-agent OR added a third one
  • You can demonstrate the manager routing different inputs to different sub-agents
  • You can describe one thing you'd change to make this pattern more robust for production

Hints

Hint 1: Understanding the supervisor pattern

The Manager Agent's system prompt is where the routing logic lives. It describes each sub-agent's capabilities and tells the manager when to delegate to each one. The sub-agents are connected as Agent Tool nodes — from the manager's perspective, each sub-agent is just another tool it can call.

Hint 2: Adding a third sub-agent
  1. Add a new AI Agent node. Give it a focused system prompt (e.g., "You are a ResearchAgent. You find and summarize information about any topic.")
  2. Wrap it in an Agent Tool node (or connect it as a tool to the manager)
  3. Update the Manager Agent's system prompt to include the new sub-agent
  4. Give the tool a clear name and description
Hint 3: Testing multi-agent routing

Key test inputs should be unambiguous:

  • "Draft a follow-up email to the client about the project deadline" → EmailAgent
  • "How many rows are in the sales spreadsheet from last month?" → DataAgent
  • "Find out what our competitors announced this week" → ResearchAgent

Watch the execution to see the manager's reasoning and which sub-agent it calls.

Step-by-step guide (optional)
  1. Import workflow #7158 from n8n.io/workflows.
  2. Identify the three layers: chat/trigger input, Manager Agent, sub-agent tools.
  3. Read the Manager Agent's system prompt — this is where routing logic is defined.
  4. Click on each sub-agent tool. Read their system prompts.
  5. Test with an email-related request. Watch the execution trace.
  6. Test with a data-related request. Verify correct routing.
  7. Add a third sub-agent:
    1. Add a new AI Agent node with a clear specialty in its system prompt.
    2. Connect your LLM model to it.
    3. Connect it as a tool to the Manager Agent (follow the existing pattern).
    4. Update the Manager's system prompt to include the new sub-agent.
  8. Test with an input matching the new sub-agent's specialty.
  9. Test with an ambiguous input. If it routes wrong, refine the tool descriptions.
  10. Think about production: what happens if the manager picks the wrong agent? How would you add a fallback?

Wrap-Up

You've now built the three patterns that make AI workflows actually useful in production: structured output parsing that turns AI responses into actionable data (Exercise 1), an agent with tools that can take real actions autonomously (Exercise 2), and a multi-agent system where a supervisor coordinates specialized agents (Exercise 3). And you did it by standing on the shoulders of the n8n community — importing templates, studying their patterns, and then customizing them for your own needs. That's exactly how this is done in practice.
