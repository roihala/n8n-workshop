Lab 1: Data Flow, Context, and LLM-Powered Building

Import broken workflows, trace data through nodes, fix bugs, and use LLMs to generate and debug workflows

Skills practiced: Tracing data through nodes, fixing broken expressions, debugging Code node JavaScript, using LLMs to generate and debug workflows

Prerequisites: n8n running locally, Lesson 1 talk completed. For exercises 3-4: access to an LLM (ChatGPT, Claude, Gemini, or any other)

The Scenario

Your colleague just left the company, and they left behind a collection of n8n workflows that the team depends on. Problem is, none of them work. Some have broken expressions, some have wrong API endpoints, some have JavaScript bugs in Code nodes. Nobody documented anything. Your job: import each workflow, figure out what's wrong, and fix it.

Pick the exercises that interest you. They're independent and you can do them in any order.

1

Fix the Data Pipeline

Approachable Data flow, expressions, HTTP requests

The Challenge

Import exercise_1_data_flow.json from the workflows/ folder. This workflow has three branches, and each one is broken in a different way. Your job is to find the bug in each branch, fix it, and get all three producing correct output.

⬇ Download exercise_1_data_flow.json

The three branches are:

  1. Split into Items — Fetches album data from JSONPlaceholder and splits the response into individual items. But the split isn't working.
  2. Data Scraping — Fetches a random Wikipedia page and extracts the article title. But the extraction comes back empty.
  3. Handle Pagination — Fetches a GitHub user's starred repos page by page. But the page tracking is broken and the workflow errors out.

Acceptance Criteria

  • Branch 1 produces individual album items (not a single blob)
  • Branch 2 extracts the article title from a random Wikipedia page
  • Branch 3 fetches at least one page of starred repos without erroring
  • You can explain what was wrong with each branch and why your fix works

Hints

Hint 1: How to diagnose any broken node

Click on the node that comes before the broken one. Look at its output. What fields does it actually produce? Now click on the broken node and look at what field name it's trying to reference. Is there a mismatch?

This "inspect the previous node's output" technique solves 80% of n8n bugs.

Hint 2: Branch 1 — the split isn't working

The Item Lists node is trying to split on a field that doesn't exist. Run the HTTP Request node by itself (click "Test step") and look at its output. What is the actual field name that contains the array of albums? The Item Lists node needs to reference that exact name.

Hint 3: Branch 2 — empty extraction

The HTML Extract node uses a CSS selector to find elements in the Wikipedia page. The current selector is looking for an element that doesn't exist. Open any Wikipedia article in your browser, right-click the article title, and "Inspect" it. What is the actual id attribute of the heading element?

Hint 4: Branch 3 — pagination errors out

Look at the error message carefully. It mentions a node name that it can't find. Now look at the actual names of the nodes on the canvas. Do they match what the expressions are referencing? In n8n, when you reference a node by name in an expression like $node["SomeName"], the name must match exactly.

2

Debug the Code Pipeline

Challenging Code nodes, JavaScript debugging, cross-node references

The Challenge

Import exercise_2_code_debugging.json from the workflows/ folder. This is a data processing pipeline that takes a list of users, enriches each one with calculated fields and external API data, computes an aggregate summary, and generates a CSV file.

⬇ Download exercise_2_code_debugging.json

It has four bugs across the pipeline. The workflow will fail at the very first processing step, and as you fix each bug, the next one will surface. Work through them one at a time from left to right.

The pipeline:

Sample Data → Split Out → Process Each User → Fetch External Data → Calculate Average → Create CSV File

Acceptance Criteria

  • The Split Out node produces 4 individual user items
  • The "Process Each User" Code node produces fullName and age for each user
  • The "Fetch External Data" Code node enriches each user with gender prediction data from an external API
  • The "Create a Binary File" Code node generates a downloadable CSV file
  • You can explain each bug and what caused it

Hints

Hint 1: The pipeline dies immediately

The Split Out node can't find the field it's looking for. Click on the "1. Sample Data" node, run it, and look at what field name the array of users is stored under. Now check what name the Split Out node is configured to split on. They need to match.

Hint 2: The Code node produces "undefined undefined"

Look at the JavaScript code in "3. Process Each User". It's trying to access fields on the user object. Now look at the actual data coming in from the previous node. What are the exact field names? JavaScript is case-sensitive, and the naming convention matters: firstName is not the same as first_name.

Hint 3: The external API call fails

The Code node is calling an API URL. Look at the URL carefully. Try opening it in your browser. Does it return data, or an error? Sometimes the path in the URL is wrong. Compare it to what the API documentation says the correct endpoint is (the API is genderize.io).

Hint 4: The CSV file node can't find its data

The last Code node uses $("node name") to reference data from an earlier node. This is how you access data from nodes that aren't your direct predecessor. But the name inside the quotes must match the exact name of the node on the canvas. Check if they match.

3

Generate a Workflow with an LLM

Creative LLM workflow generation, prompt engineering, the generate → test → fix loop

The Challenge

Use an LLM (ChatGPT, Claude, Gemini, or whichever you prefer) to generate a complete n8n workflow as JSON. Then import it into n8n, test it, find the issues, and prompt the LLM to fix them.

Here's the spec for the workflow you should ask the LLM to generate:

"Build an n8n workflow that:
  1. Starts with a Manual Trigger
  2. Fetches a random programming joke from https://v2.jokeapi.dev/joke/Programming?blacklistFlags=nsfw,religious,political,racist,sexist,explicit&type=single
  3. Fetches a random inspirational quote from https://zenquotes.io/api/random
  4. Combines both results into a single item using a Merge node
  5. Produces a final output with three fields: joke (the joke text), quote (the quote text), author (the quote author), and timestamp (current time)

Return only valid n8n workflow JSON that can be imported directly. No explanation, just the JSON."

The goal isn't to get it perfect on the first try. The goal is to practice the loop: describe → generate → import → test → find issue → ask LLM to fix → re-import.

Acceptance Criteria

  • You prompted an LLM and received n8n workflow JSON
  • The JSON imports into n8n (even if it doesn't run perfectly yet)
  • You identified at least one issue with the generated workflow
  • You prompted the LLM with the specific issue and got an updated version
  • The final workflow runs and produces output with joke, quote, author, and timestamp
  • You saved the final workflow JSON

Hints

Hint 1: Structuring your prompt

The more specific you are, the better. Include:

  • Exact node types you want (n8n-nodes-base.httpRequest, n8n-nodes-base.merge, n8n-nodes-base.set)
  • Exact API URLs
  • That you want the trigger to fan out to both HTTP requests in parallel
  • The exact output field names and where they come from

Generic prompts get generic (broken) results. Specific prompts get closer to working.

Hint 2: Common issues with LLM-generated workflows

The most common problems:

  • Wrong node type names — the LLM might invent node types that don't exist
  • Broken expressions — the {{ }} syntax might be slightly off
  • Wrong Merge configuration — the Merge node has multiple modes and the LLM often picks the wrong one
  • Missing connections — some nodes might not be wired up
  • Wrong API response field names — the LLM might guess $json.quote when the API returns $json.q

When you find an issue, tell the LLM exactly what went wrong.

Hint 3: The zenquotes API gotcha

The zenquotes.io API returns fields with single-letter keys: q for the quote text and a for the author. Most LLMs don't know this and will generate expressions like $json.quote or $json.text. If your generated workflow has this problem, tell the LLM: "The zenquotes API at /api/random returns an array with objects having 'q' (quote) and 'a' (author) fields, not 'quote' and 'text'."

4

Use an LLM to Debug a Workflow

Investigative LLM-assisted debugging, exporting workflows, describing problems to an LLM

The Challenge

Import exercise_4_llm_debugging.json from the workflows/ folder. This workflow is supposed to fetch a random quote and a programming joke, combine them, and produce a formatted "daily message" output.

⬇ Download exercise_4_llm_debugging.json

When you run it, it won't crash with an obvious error. It will produce output, but the output will be wrong. Something is off, and the bugs are subtle enough that reading through the nodes manually will take a while.

Instead of debugging manually, try this approach:

  1. Run the workflow and observe the output
  2. Click through the nodes and note what looks wrong (what data did you expect vs. what you got?)
  3. Export the workflow JSON (three-dot menu → Download)
  4. Paste the JSON into an LLM along with a description of the symptoms
  5. Ask the LLM to identify the bugs
  6. Apply the fixes the LLM suggests
  7. Run again and verify

Acceptance Criteria

  • You ran the workflow and identified that the output is wrong (describe what you expected vs. what you got)
  • You exported the workflow JSON and pasted it into an LLM
  • The LLM identified at least one bug you hadn't found yet
  • You fixed all the issues and the workflow now produces a clean daily message with one quote and one joke
  • You can explain what was wrong and how the LLM's analysis helped

Hints

Hint 1: What to look for when the output is "wrong but not broken"

Click on each node after execution and count how many items it produces. Does the number make sense? If a node that should produce 1 item produces 50, something is wrong upstream. Also check: are the field values what you expect, or are they empty/undefined?

Hint 2: How to prompt the LLM for debugging

A good debugging prompt structure:

Here is an n8n workflow JSON:
[paste the JSON]

When I run this workflow, I expect it to:
- Fetch one random quote
- Fetch one programming joke
- Merge them into a single output with fields: quote, author, joke, dailyMessage

Instead, what happens is:
[describe the symptoms you observed]

What bugs can you find in this workflow?

The more specific you are about the expected vs. actual behavior, the better the LLM can help.

Hint 3: There are multiple issues

This workflow has at least 3 bugs. One is in an API URL, one is in field name references, and one is a data shape problem where the wrong number of items flows through. The LLM should be able to identify all three if you describe the symptoms well.

Wrap-Up

If you completed exercises 1 and 2, you've built the core diagnostic skill of n8n development: tracing data through nodes, reading error messages, matching field names, and understanding how expressions reference data across the workflow. These are the skills that separate someone who can click through tutorials from someone who can actually build and maintain production workflows.

If you completed exercises 3 and 4, you've practiced what is quickly becoming the most practical workflow development skill: using an LLM to generate and debug workflows. The generation loop (describe → generate → test → fix) and the debugging loop (observe symptoms → export → ask LLM → apply fixes) are patterns you'll use every day. The key insight is that LLMs are good at n8n workflows because every workflow is just JSON, and JSON is something LLMs understand deeply.
